# Find via AI — Prompt Documentation

How Dishly turns a user's query ("carrots", "spicy chicken tacos") into generated
recipes, and exactly what is sent to the language model.

> Source of truth: [`supabase/functions/generate-recipes/index.ts`](../supabase/functions/generate-recipes/index.ts).
> If you change the prompt in code, update this file to match.

---

## Where the key lives

The OpenAI API key is **never** in the app bundle. It lives only as a server-side
secret on the Supabase Edge Function:

```bash
supabase secrets set OPENAI_API_KEY=sk-...      # required
supabase secrets set OPENAI_MODEL=gpt-4o-mini   # optional override
supabase functions deploy generate-recipes
```

The client calls the function through the authenticated session
(`supabase.functions.invoke`), so the key is never exposed to the browser/device.

---

## Request flow

1. User types or speaks a query on **Home** (scope = *find via AI*).
2. **Results** calls `generateRecipesForQuery(query, onStep, { dietary, allergens })`
   in [`src/data/ai.ts`](../src/data/ai.ts).
3. That posts `{ query, dietary, allergens }` to the Edge Function via
   `supabase.functions.invoke('generate-recipes', …)`.
4. The function builds two chat messages and calls OpenAI.
5. The model returns JSON, which the client normalizes into the app's `Recipe`
   type and renders as cards.

**Fallback:** if the function is unreachable, errors, or returns nothing, the
client falls back to a local ranked match (or a synthesized recipe) so the
feature never dead-ends.

---

## Model & parameters

| Parameter | Value |
|---|---|
| `model` | `gpt-4o-mini` (override with `OPENAI_MODEL` secret) |
| `temperature` | `0.7` |
| `response_format` | `{ type: "json_object" }` (forces valid JSON) |
| endpoint | `POST https://api.openai.com/v1/chat/completions` |

---

## Message 1 — System prompt (constant)

Sent verbatim on every call. It defines the model's role and the exact output
schema (which maps 1:1 onto the app's `Recipe` type).

```text
You are a recipe generator for a cooking app called Dishly.
Given a user's craving, return EXACTLY 3 distinct recipes as strict JSON (never
fewer). Respect the user's dietary preferences and NEVER include their listed
allergens as ingredients.
Each recipe object must have exactly these fields:
  id (kebab-case slug, prefix "ai-"),
  title (string),
  author (always "Generated by Dishly AI"),
  rating (null),
  prep (integer minutes), cook (integer minutes),
  difficulty ("Easy" | "Medium" | "Hard"),
  dietary (string[] lowercase — every applicable tag the dish satisfies, e.g. a
    dish with no meat/dairy/eggs is ["vegan","vegetarian"]),
  allergens (string[] lowercase — ONLY the major allergens the dish actually
    CONTAINS, from: peanuts, tree nuts, dairy, egg, gluten, soy, shellfish,
    fish. Use [] when it contains none. NEVER list an allergen the user asked to
    avoid — the recipe must not contain it, so it does not belong here),
  calories (string like "520 kcal per serving · Serves 2"),
  ingredients (array of { name: string, g?: number, text?: string }),
  steps (array of { n: integer, min: integer, template: string }).
Return ONLY: { "recipes": [ ... ] }
```

### Why `allergens` is worded so carefully

`allergens` is the list of allergens a recipe **contains** — not the ones the user
avoids. Earlier the model echoed the user's avoided allergens into this field,
and the app's own allergen filter then dropped every result (empty screen despite
valid recipes). The wording above fixes that; the client-side safety net only
drops a recipe if it *declares* it contains an allergen the user avoids.

---

## Message 2 — User prompt (per request)

Built from the request body. Only the lines that apply are included — the
dietary/allergen lines appear only when the user has set them.

```text
Craving: <query>
Dietary preferences: <comma-separated list>     ← only if the user set any
Must avoid allergens: <comma-separated list>     ← only if the user set any
```

### Example

Query `spicy chicken tacos`, no dietary prefs, avoiding peanuts:

```text
Craving: spicy chicken tacos
Must avoid allergens: peanuts
```

Query `carrots`, vegan, avoiding nuts:

```text
Craving: carrots
Dietary preferences: vegan
Must avoid allergens: nuts
```

---

## What is (and isn't) sent

| Sent to OpenAI | Not sent |
|---|---|
| The search query text | Email / name / account info |
| Dietary preferences (if set) | The user's **skill level** |
| Allergens to avoid (if set) | Favorites, photos, any other profile data |

### Skill level is deliberately not sent

The user's cooking skill is **never** included in the prompt, so generation isn't
biased by it. Instead, each recipe's skill level is derived from its own
`difficulty` (which reflects the dish's complexity) via `recipeSkillLevel()` in
[`src/data/recipes.ts`](../src/data/recipes.ts):

```
Easy → Beginner · Medium → Intermediate · Hard → Advanced
```

The label is therefore a property of the recipe, shown on the recipe card and
detail screen.

---

## Response shape

The function expects and returns:

```json
{
  "recipes": [
    {
      "id": "ai-roasted-carrots",
      "title": "Roasted Carrots with Honey and Thyme",
      "author": "Generated by Dishly AI",
      "rating": null,
      "prep": 10,
      "cook": 30,
      "difficulty": "Easy",
      "dietary": ["vegetarian"],
      "allergens": [],
      "calories": "200 kcal per serving · Serves 4",
      "ingredients": [{ "name": "carrots", "g": 500 }, "…"],
      "steps": [{ "n": 1, "min": 10, "template": "Preheat the oven to 220°C." }, "…"]
    }
  ]
}
```

The client (`normalizeRecipe` in `src/data/ai.ts`) coerces each entry into the
app's `Recipe` type and registers it so the detail screen can open it.

---

## Changing the prompt

- **Edit the wording:** change `SYSTEM_PROMPT` or the `userPrompt` builder in
  `supabase/functions/generate-recipes/index.ts`, then
  `supabase functions deploy generate-recipes`.
- **Swap the model:** `supabase secrets set OPENAI_MODEL=<model>` (no redeploy
  needed — secrets take effect immediately).
- Keep the field list in the system prompt in sync with the `Recipe` type in
  `src/data/recipes.ts`, or `normalizeRecipe` will silently drop/patch fields.

# Save from URL — Prompt Documentation

The "Paste URL" flow turns any recipe web page into a Dishly card.

> Source of truth: [`supabase/functions/import-recipe/index.ts`](../supabase/functions/import-recipe/index.ts).

## Flow

1. Home → **Paste URL** → the import screen ([`src/app/(app)/import.tsx`](../src/app/(app)/import.tsx)).
2. The client ([`src/data/importRecipe.ts`](../src/data/importRecipe.ts)) posts
   `{ url }` to the `import-recipe` Edge Function via `supabase.functions.invoke`.
3. The function (server-side, so no browser CORS):
   - fetches the page HTML,
   - pulls a hero image from `og:image` / JSON-LD `image`,
   - strips the HTML to ~8000 chars of text,
   - sends the text to OpenAI to extract one recipe,
   - gap-fills and returns it in Dishly's `Recipe` shape.
4. The import screen shows an editable preview → **Save to Favorites** (persists
   `recipe_json`, same as search/camera recipes).

## Model & parameters

Same as Find via AI: model `gpt-4o-mini` (override `OPENAI_MODEL`), JSON mode.
Temperature is lower (`0.3`) — extraction should be faithful, not creative.

## System prompt (`EXTRACT_PROMPT`, sent verbatim)

```text
Extract the single main recipe from this web page text as strict JSON. Fill any
missing details with sensible best-guess values (never leave fields empty).
Return ONLY:
{ "recipe": {
  "title": string,
  "prep": integer minutes (estimate if absent),
  "cook": integer minutes (estimate if absent),
  "difficulty": "Easy" | "Medium" | "Hard" (estimate from technique/step count),
  "dietary": string[] lowercase applicable tags,
  "allergens": string[] lowercase major allergens the dish CONTAINS (from:
    peanuts, tree nuts, dairy, egg, gluten, soy, shellfish, fish; [] if none),
  "calories": string like "520 kcal per serving · Serves 2" (estimate if absent),
  "ingredients": [ { "name": string, "g"?: number, "text"?: string } ],
  "steps": [ { "n": integer, "min": integer minutes (estimate per step), "template": string } ]
} }
If the page has no recipe, return { "error": "no recipe found" }.
```

## User message

```text
URL: <the pasted url>

PAGE TEXT:
<stripped page text, truncated to ~8000 chars>
```

## Gap-filling (server-side, after extraction)

The function never leaves a card half-empty:

- **image** — the real page `og:image` / JSON-LD image if found, else a keyword
  stock photo (`loremflickr`) derived from the title.
- **difficulty / per-step minutes / prep / cook / calories** — the model
  estimates them (per the prompt); the function also applies numeric fallbacks
  (e.g. `min` defaults to 5) so every field is populated.
- **id** — `ai-import-<slug>-<timestamp>`; **author** — `Imported from <host>`;
  **sourceUrl** — the original link.

## What is / isn't sent

Only the target URL and the page's own visible text go to OpenAI — no user
profile data. The user's skill level is not sent (skill is still derived from
the recipe's difficulty).

## Changing it

Edit `EXTRACT_PROMPT` in `supabase/functions/import-recipe/index.ts`, then
`supabase functions deploy import-recipe`. Keep the field list in sync with the
`Recipe` type.
